# 第一章 习题与思考

## 习题 1：智能体类型判定

请分析以下四个 case 中的主体是否属于智能体，如果是，那么属于哪种类型的智能体（可以从多个分类维度进行分析），并说明理由：

### Case A：超级计算机
**描述**：一台符合冯·诺依曼结构的超级计算机，拥有高达每秒 2EFlop 的峰值算力。
**分析**：
- **是否为智能体**：否（或极其有限）。
- **理由**：虽然它拥有强大的计算能力，但如果它只是被动等待指令执行计算任务，缺乏自主性（Autonomy）和持续感知-行动循环（Perception-Action Loop），它更像是一个强大的工具（Tool）而非智能体。

### Case B：自动驾驶系统
**描述**：特斯拉自动驾驶系统在高速公路上行驶时，突然检测到前方有障碍物，需要在毫秒级做出刹车或变道决策。
**分析**：
- **是否为智能体**：是。
- **类型**：
    - **决策架构**：基于模型的反射智能体（Model-Based Reflex Agent）。它需要维护车辆当前状态和周围环境模型。
    - **反应性**：反应式智能体（Reactive Agent）。在紧急避障场景下，必须在毫秒级做出反应，没有时间进行深思熟虑的规划。
- **理由**：具备感知（摄像头/雷达）、行动（刹车/转向）和自主决策能力。

### Case C：AlphaGo
**描述**：AlphaGo在与人类棋手对弈时，需要评估当前局面并规划未来数十步的最优策略。
**分析**：
- **是否为智能体**：是。
- **类型**：
    - **决策架构**：学习型智能体（Learning Agent）+ 基于效用的智能体（Utility-Based Agent）。
    - **反应性**：规划式智能体（Deliberative Agent）。它通过蒙特卡洛树搜索（MCTS）进行深度规划，预测未来多步棋局。
- **理由**：它通过自我对弈学习策略，并在每一步决策时进行前瞻性规划以最大化胜率（效用）。

### Case D：智能客服
**描述**：ChatGPT 扮演的智能客服在处理用户投诉时，需要查询订单信息、分析问题原因、提供解决方案并安抚用户情绪。
**分析**：
- **是否为智能体**：是。
- **类型**：
    - **决策架构**：LLM 驱动的智能体（LLM-based Agent）。
    - **反应性**：混合式智能体（Hybrid）。既有即时的语言回复能力，也有调用工具查询订单的规划能力。
- **理由**：具备自然语言理解、工具使用（查询订单）和情感交互能力。

## 习题 2：智能健身教练 PEAS 分析

假设你需要为一个"智能健身教练"设计任务环境。

**PEAS 描述**：

| 要素 | 描述 |
| :--- | :--- |
| **Performance (性能度量)** | 用户的健康指标改善程度（体脂率下降、耐力提升）、用户满意度评分、训练计划的完成率、动作纠正的准确性。 |
| **Environment (环境)** | 用户的身体状况（动态变化）、健身场地（健身房/家庭）、可用的器械、即时的运动数据流。 |
| **Actuators (执行器)** | 语音指导（TTS）、视觉反馈（屏幕显示动作演示）、智能手表的震动提醒、调整跑步机/单车的阻力参数。 |
| **Sensors (传感器)** | 智能手表/手环（心率、加速度计）、摄像头（捕捉动作姿态）、麦克风（接收用户反馈）、体脂秤数据接口。 |

**环境特性分析**：
- **部分可观察 (Partially Observable)**：无法完全获知用户的所有生理状态（如肌肉酸痛的具体程度）。
- **随机性 (Stochastic)**：用户的状态波动大，可能突发不适，训练效果具有不确定性。
- **动态性 (Dynamic)**：在智能体决策（如生成建议）的同时，用户的心率和体力仍在实时变化。
- **连续性 (Continuous)**：运动数据是连续的时间序列，而非离散的步骤。

## 习题 3：Workflow vs Agent

某电商公司正在考虑两种方案来处理售后退款申请。

**优缺点分析**：

| 方案 | 优点 | 缺点 | 适用场景 |
| :--- | :--- | :--- | :--- |
| **方案 A (Workflow)** | 1. **确定性高**：规则明确，结果可预测。<br>2. **成本低**：不需要昂贵的大模型推理。<br>3. **易审计**：每一笔退款的原因都清晰可查。 | 1. **僵化**：无法处理边缘情况（如虽超期但属于质量召回）。<br>2. **维护难**：规则变多后，流程图会变得极其复杂。<br>3. **体验差**：对特殊商品一刀切，缺乏人情味。 | 规则明确、高频重复、风险可控的标准业务流程。 |
| **方案 B (Agent)** | 1. **灵活性强**：能理解复杂的退款理由和上下文。<br>2. **用户体验好**：能像人一样综合判断，更有温度。<br>3. **泛化能力**：能处理未预见的特殊情况。 | 1. **不确定性**：可能产生幻觉或误判。<br>2. **成本高**：Token 消耗和推理延迟。<br>3. **黑盒**：决策逻辑较难完全解释。 | 复杂、非标准、需要综合判断或情感交互的场景。 |

**选择建议**：
如果我是负责人，我会倾向于 **方案 C (融合方案)**：
- **第一层（Workflow）**：用规则快速过滤掉 80% 的简单情况（如明显合规的小额退款自动秒批，明显违规的直接驳回）。
- **第二层（Agent）**：将处于“灰色地带”或复杂的申请交给 Agent 进行初步分析和建议，生成一份摘要。
- **第三层（人工）**：对于高金额或 Agent 拿不准的单子，由人工最终复核。

这样既保证了效率和合规，又提升了处理复杂问题的灵活性。

## 习题 4：智能旅行助手的功能扩展

**设计思路**：

1.  **记忆功能 (Memory)**：
    - **实现**：引入一个持久化存储（如 JSON 文件或向量数据库）。
    - **逻辑**：在 `SimpleAgent` 类中增加 `self.user_profile`。每次对话结束时，让 LLM 总结用户的偏好（"用户喜欢穷游"、"用户对古迹感兴趣"）并更新到 profile 中。下次对话开始时，将 profile 作为 System Prompt 的一部分注入。

2.  **门票售罄处理 (Exception Handling)**：
    - **实现**：修改 `get_attraction` 工具，使其返回结果中包含“状态”字段。
    - **逻辑**：当 Observation 返回 "故宫（门票已售罄）" 时，LLM 需要在 Thought 中识别这一状态，并触发新的 Action，例如调用 `search_alternatives(original_spot="故宫")` 或重新调用推荐工具并附加约束 `exclude=["故宫"]`。

3.  **反思与调整 (Self-Reflection)**：
    - **实现**：维护一个短期记忆 `rejected_history`。
    - **逻辑**：当检测到用户连续输入“不喜欢”、“换一个”时，触发反思机制。
    - **Prompt 调整**：发送一个特殊的 Prompt 给 LLM："用户已经连续拒绝了 3 次推荐。请反思之前的推荐策略有什么问题，并尝试提出一个完全不同风格的建议。"

## 思考题：系统 1 与 系统 2 的协同

**应用场景**：医疗诊断助手

**分工协同**：

- **系统 1 (快速直觉 / 亚符号 AI)**：
    - **任务**：医学影像分析（X光片、CT图）。
    - **角色**：利用深度学习模型（CNN），快速识别影像中的异常斑点或肿瘤区域。它能一眼看出“这里看起来不对劲”，但很难解释为什么。
    - **输出**：`Observation: 肺部左下叶发现 3mm 结节，置信度 98%。`

- **系统 2 (慢速推理 / 符号 AI / LLM)**：
    - **任务**：综合诊断与治疗建议。
    - **角色**：结合系统 1 的发现、病人的电子病历（历史过敏史、家族病史）以及医学指南（知识库），进行逻辑推理。
    - **过程**：
        1. 接收系统 1 的观察结果。
        2. 查询知识库：该类结节的诊疗指南。
        3. 检查病历：病人是否有吸烟史？
        4. 推理：结合以上信息，判断结节良恶性概率，制定后续检查计划。
    - **输出**：`建议进行增强 CT 扫描，并预约呼吸科专家门诊。`

**协同方式**：系统 1 作为极其敏锐的“眼睛”，负责从非结构化数据（图像）中提取关键特征；系统 2 作为理性的“大脑”，利用这些特征结合规则和知识进行最终决策。

## 思考题：局限性分析

1.  **为什么会产生幻觉？**
    - **本质**：LLM 本质上是基于概率的“下一个词预测器”。它并不真正理解事实，只是在模仿训练数据中的语言模式。
    - **数据源**：训练数据中本身可能包含错误或矛盾信息。
    - **压缩损耗**：模型将海量知识压缩到参数中时，丢失了精确的索引，导致“张冠李戴”。

2.  **没有最大循环次数限制会怎样？**
    - **死循环**：如果模型陷入思维怪圈（例如反复调用同一个报错的工具），程序将永远运行下去，消耗大量 Token 和费用。
    - **资源耗尽**：内存溢出或 API 额度超限。

3.  **如何评估智能程度？**
    - **准确率不够**：仅看最终结果（准确率）无法反映过程的合理性。
    - **多维评估指标**：
        - **效率**：达成目标用了多少步骤？调用了多少次工具？
        - **鲁棒性**：面对干扰信息或工具报错时，能否自我修正？
        - **安全性**：是否会执行危险指令？
        - **解释性**：Thought 过程是否逻辑自洽？
